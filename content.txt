Product Features and Measurement Framework ##### The Eight Dimensions of Intelligence VerifiedSignal evaluates every document through eight distinct system-wide intelligence lenses: 1. Logical Fallacy Rating: Measures manipulative reasoning. Indicators: Per-fallacy breakdown of ad hominem, straw man, false dichotomy, and slippery slope, mapped to specific text triggers. 2. Factuality Confidence: Measures the reliability of claims. Indicators: Internal consistency, citation signals, and cross-referenced factual claims with a detailed rationale. 3. AI Generation Probability: Measures machine authorship likelihood. Indicators: Multi-model assessment of linguistic patterns vs. known LLM signatures with a specific model guess included. 4. Pseudoscience Indicators: Measures adherence to scientific rigor. Indicators: Detection of unfalsifiable claims, anecdotal evidence, "appeal to nature," and absence of peer review. 5. Fictional Content Likelihood: Measures intent and genre. Indicators: Distinguishes reported fact from narrative, satire detection, or speculation presented as journalism. 6. Source Provenance: Measures document origin history. Indicators: Domain reputation, WHOIS history, canonical URL verification, and archive.org presence. 7. Semantic Search: Measures deep content relevance. Indicators: Vector-based similarity (kNN) and hybrid retrieval across massive document collections. 8. Collection Analytics: Measures aggregate quality shifts. Indicators: Trend dashboards showing fluctuations in factuality or fallacy frequency across sources over time. ##### The "Agentic" Workflow 1. Submit: User uploads file (PDF, Word, HTML) or URL. System performs automated fetching, text extraction, and cleaning. 2. Investigate: The system verifies source provenance (publication dates, author identity, and domain history) before analysis begins. 3. Analyze: A battery of AI models simultaneously scores the document across the eight dimensions. 4. Collect: Users save documents to collections for side-by-side comparison and trend visualization. ##### User Interface Standards * Review UI: Features a side-by-side view for Human-in-the-Loop (HITL) verification, allowing users to correct low-confidence AI extractions. * Document Explorer: Implements "Suspicious Documents" card logic, automatically flagging items with High AI probability (>0.7) and Low Factuality (<0.4). * Real-Time Feedback: Scorecards populate via Server-Sent Events (SSE) to show field-level progress during the analysis phase. -------------------------------------------------------------------------------- #### 3. High-Level Technical Architecture ##### Core Tech Stack * AWS Amplify: Frontend hosting and global edge delivery. * Amazon S3: Object storage using Presigned URLs for direct-to-S3 uploads, minimizing Lambda execution costs. * Amazon Cognito: User authentication with "Sign In with AWS" for Marketplace native integration. * Amazon Textract: High-accuracy OCR for tables, forms, and document geometry. * Amazon Bedrock: LLM reasoning (Claude 3.5/Llama 3) for "Agentic" sanity checks and scoring. ##### Database and Search Strategy !IMPORTANT Postgres is the System of Record. Derived search state is expendable. * PostgreSQL: Canonical truth (users, permissions, final scores, billing). * Amazon OpenSearch: Disposable analytics replica for full-text search, kNN vector similarity, and dashboard aggregations. -------------------------------------------------------------------------------- #### 5. Data Plane and Intelligence Pipeline ##### The 8-Stage Ingestion Pipeline 1. Intake: Client submits file/URL; FastAPI creates "queued" record in Postgres. 2. Acquisition: Worker fetches bytes, computes content hashes, and checks for duplicates. 3. Extraction: Textract/Worker extracts plain text and structure; publishes progress to Redis. 4. Enrichment: System computes metadata, topical tags, and initial quality flags. 5. LLM Scoring: Worker calls Bedrock for structured analysis via the Converse API. 6. Canonical Persistence: Final validated scores are written to the authoritative PostgreSQL layer. 7. Search Indexing: Document is indexed into OpenSearch for retrieval and analytics. 8. Completion Notification: Final "done" event is pushed to the frontend via SSE. ##### LLM Implementation & Prompt Engineering * The "Auditor" System Prompt: Uses strict XML tagging (, ) and JSON schema enforcement . It forces math validation (e.g., Total = Subtotal + Tax). * Few-Shot Strategy: Includes "Success," "Graceful Failure" (teaching the model to return null for missing data), and an "Edge Case" example (a messy, low-quality document) to prevent hallucinations. * Model Settings: Temperature is set to 0.0 for deterministic, "boring auditor" consistency. -------------------------------------------------------------------------------- #### 6. Streaming and Real-Time UX ##### Server-Sent Events (SSE) Design VerifiedSignal uses SSE for one-way status streaming. !IMPORTANT Nginx must be configured with the X-Accel-Buffering: no header to prevent buffering from breaking the real-time stream. Sample Event Contract (JSON): json { "event": "stage", "document_id": "doc_123", "stage": "extract_text", "status": "running", "timestamp": "2026-03-26T22:11:10Z", "progress": 35 }  ##### Incremental LLM Scoring The system utilizes partial JSON token parsing . As tokens stream from Bedrock, a worker identifies completed fields and updates the UI before the full LLM response is finished. Field State Machine: unseen $\rightarrow$ in_progress $\rightarrow$ complete $\rightarrow$ validated $\rightarrow$ persisted. -------------------------------------------------------------------------------- #### 7. AWS Marketplace & Integration Plumbing ##### The "Plumbing" Layer When a user subscribes, AWS sends a POST with an x-amzn-marketplace-token. The backend exchanges this for a CustomerIdentifier to link the AWS billing account to the VerifiedSignal subscription. ##### Metering and Billing A scheduled BatchMeterUsage Lambda runs hourly, aggregating document counts from Postgres and reporting them to the AWS Marketplace Metering Service. This enables "Zero Idle Cost" serverless billing. -------------------------------------------------------------------------------- #### 8.
